# Reverse Engineering — Olympia Astrology activation mechanism

Static analysis of `Olympia.exe` (Qt6 / C++ native, MinGW-w64, x86-64, ~194 MB) using Ghidra.

## Target

- Binary: `Olympia.exe` (Olympia Astrology desktop app)
- Goal: understand the license-activation flow, starting from the in-binary string `"The software is activated !"`.

## Methodology

1. **String triage** — locate the strings related to the activation flow inside the `.rdata` cluster around `0x14b060000`.
2. **Xref pivot** — follow cross-references from those strings back into `.text`.
3. **Dispatcher recognition** — identify the `qt_static_metacall` jump table that routes to the activation window's slots.
4. **Slot decompilation** — decompile the two relevant slots (validation + reset) and read the algorithm.

## Key strings (`.rdata`)

| Address | String |
|---|---|
| `0x14b061901` | `"Invalid password.\n"` |
| `0x14b060f08` | `"Limit of trials of password inputs."` |
| `0x14b0611d0` | `"Too many passwords have been prompted."` |
| `0x14b060d06` / `0x14b0a0d83` | `"IA/Password"` (QSettings key) |
| `0x14b060120` | `"2- Activate OLYMPIA with the Password"` |
| `0x14b060148` | `"Activate OLYMPIA on a new computer"` |

The strings are all clustered, indicating they belong to a single translation unit (the activation widget).

Note: `"2activated()"` strings appearing dozens of times in the binary are **Qt signal signatures** (the `'2'` prefix is `QMETHOD_CODE` for `SIGNAL(...)`), unrelated to license activation.

## MOC dispatcher

`FUN_141261e80` is the `qt_static_metacall` generated by Qt's MOC for the activation widget. It is a 16-entry switch dispatching to slots via tail jumps:

```asm
141261e80: CMP  EDX, 0xF
141261e83: JA   ret
141261e89: LEA  R8, [0x14b30f700]   ; jump table in .rdata
141261e92: MOV  EDX, EDX
141261e99: JMP  RAX                  ; tail call to slot N
```

Because the slots are reached by indirect `JMP` (not `CALL`), Ghidra's autoanalyzer did not create functions at the targets. Slots had to be defined manually (`F`) at each jump target before decompilation.

Two slots are relevant:

| Slot | Address | Role |
|---|---|---|
| 2 | `0x1404e2920` | "Activate with the Password" handler |
| 4 | `0x1404e0900` | "Activate on a new computer" handler (reset) |

## Slot 2 — password validation (`0x1404e2920`)

```cpp
QSettings s("Olympia", <app>);
int tries = s.value("Installation", 0).toInt();

if (tries >= 5)        showBlockedDialog();
else if (tries == 4)   showLastTryWarning();

QString input = QInputDialog::getText(this, "Password", ...);

// Three candidate strings stored as widget members:
//   this + 0x120 / 0x128  (ptr / length)
//   this + 0x138 / 0x140
//   this + 0x150 / 0x158
// Comparison via QtPrivate::equalStrings()

if (input == cand0 || input == cand1 || input == cand2) {
    s.setValue("Password", input);
    s.setValue("IA/Password", <flag>);
    QMessageBox::information(this, "Olympia Astrologie",
                             "The software is activated !");
    // ...enable Start buttons
} else {
    s.setValue("Installation", tries + 1);
    QMessageBox::critical(this, "Password",
        "Invalid password.\nOlympia Astrology cannot be Activated.\n...");
}
```

**No cryptography.** The check is a plain `QString` equality against three reference strings stored as widget data members.

UI strings are switched by `*(int*)(this + 0x250)` (language: `0=FR`, `1=EN`, `2=ES`).

## Slot 4 — "Code Reset Olympia" (`0x1404e0900`)

A second activation path titled `"Code Reset Olympia"` with prompt `"Code ?"`:

```cpp
QString input = QInputDialog::getText(this, "Code Reset Olympia", "Code ?", ...);

QDate today = QDate::currentDate();
int expected = (today.day() - today.month() + today.year()) * 0x10f;   // 0x10f == 271

if (input == QString::number(expected)) {
    QSettings s("Olympia", <app>);
    s.setValue("Password",    <input value>);
    s.setValue("IA/Password", <input value>);
    QMessageBox::information(this, "Mot de passe", ...);
}
```

The expected code is **derived locally and deterministically** from the current date:
`expected = (day - month + year) * 271`.
Validation writes the same QSettings keys as the regular activation path.

## Persistent state

All activation state lives in `QSettings("Olympia", <app>)`, i.e. under
`HKCU\Software\Olympia\...` on Windows (or in an `.ini` if forced by org settings).

| Key | Role |
|---|---|
| `Installation` | Failed-attempt counter, lockout at 5 |
| `Password` | The validated password (stored in plaintext) |
| `IA/Password` | Activation marker for the EgerIA module |

## Security observations

- No cryptography, no signature, no machine binding. The check reduces to string equality against literals embedded in the binary.
- The activated state is a QSettings value, so it is copyable between machines and editable by the user.
- The lockout counter is itself stored in the user-writable registry — resettable.
- The "Reset" path is a deterministic time-based function whose secrecy relies on obscurity (`(day - month + year) * 271`).
- The validated password is persisted in cleartext under HKCU.
- Support contact `olympia.astrologie@gmail.com` is embedded as plain string in `.rdata`.

## Tooling notes

- Ghidra MCP for navigation, xrefs, and decompilation.
- The 5-second decompiler timeout is easy to hit on large MinGW functions; retry or fall back to disassembly.
- For Qt binaries, expect MOC dispatchers (`qt_static_metacall`) to obscure the slot graph — define slot functions manually at jump-table targets before decompiling.
- The `'1'`/`'2'` Qt signal/slot prefix in strings can be a noisy false positive when grepping for `activate`.
